¿Por qué esta clase?
Un GeoServer sin asegurar es un GeoServer de otro
🤖
Los bots no descansan
Cualquier GeoServer expuesto en una red recibe intentos automáticos con admin/geoserver en minutos. Es la credencial por defecto más conocida del mundo GIS.
🗺️
No todos los datos son públicos
Un mapa base puede ser abierto, pero las capas de catastro, salud o infraestructura crítica de una IDE municipal suelen requerir autorización.
Nivel 1Cuenta admin — contraseña propia (hoy)
Nivel 2Datos — reglas por workspace y capa (hoy)
Nivel 3Servicios — quién puede usar WFS, WMS (hoy)
Nivel 4Base de datos — geoserver_user solo lectura (✓ Clase 3)
✓
El nivel 4 ya lo construiste sin saberlo: aunque alguien comprometa GeoServer, geoserver_user no puede modificar ni borrar datos en PostGIS. Hoy cerramos los otros tres niveles.
Paso 1 de 5
Primero,
la cuenta admin
Dos minutos que evitan el 90% de los ataques
Cuenta de administración
Cambiar la contraseña de admin
1
Seguridad → Usuarios, Grupos, Roles
Pestaña Usuarios/Grupos → servicio default → clic en el usuario admin.
2
Escribir la contraseña nueva y confirmar
Larga y única — mínimo 12 caracteres, sin reutilizar la de otro servicio. Es la llave de todo el servidor.
3
Guardar y volver a entrar
Cierra sesión y vuelve a iniciar con la contraseña nueva antes de cerrar la ventana — así confirmas que quedó bien guardada.
🚨
Nunca en producción con admin/geoserver
GeoServer muestra una advertencia en la página de bienvenida mientras la contraseña por defecto siga activa. Esa advertencia debe desaparecer hoy, antes de seguir con el resto de la clase.
🔑
La master password es otra cosa
Protege el almacén de claves interno (keystore) de GeoServer — no la interfaz web, sino los secretos que GeoServer guarda cifrados, como la contraseña del Store PostGIS. Se cambia en Seguridad → Contraseñas. Anótala en un lugar seguro: sin ella no se recuperan esas credenciales guardadas.
Paso 2 de 5
Usuarios y roles
El modelo de seguridad de GeoServer
Modelo de seguridad
Usuarios → Roles → Reglas: tres piezas, una idea
👤
Usuarios
Personas o sistemas que se autentican: admin · editor_gis
🎭
Roles
Etiquetas de permiso: ADMIN · ROLE_EDITOR · ROLE_ANONYMOUS
🔐
Reglas de acceso
Qué rol puede leer (r) o escribir (w) en cada workspace y capa
💡
Un ejemplo concreto
El usuario editor_gis se autentica con su contraseña. Tiene asignado el rol ROLE_EDITOR. Una regla dice que ROLE_EDITOR puede leer v_altiplano. Resultado: editor_gis puede ver esa capa — pero no porque la regla lo mencione a él, sino a su rol.
💡
Las reglas apuntan a roles, no a personas
Si mañana entran 5 editores nuevos al proyecto, solo les asignas el rol ROLE_EDITOR — las reglas no se tocan. Por eso nunca se escribe una regla para un usuario individual.
🆕
GeoServer 3: OIDC reemplaza Keycloak
Para autenticación corporativa (entrar con la cuenta institucional), la extensión OIDC soporta el estándar OAuth2/OpenID Connect: Keycloak, Auth0, Google, Azure AD, todos con la misma extensión. En este curso usamos los usuarios locales de GeoServer — suficiente para una IDE municipal pequeña.
En la interfaz de GeoServer
Crear el rol EDITOR y el usuario editor_gis
1
Seguridad → Usuarios, Grupos, Roles → pestaña Roles
Agregar nuevo rol → nombre: ROLE_EDITOR (el prefijo ROLE_ es la convención de Spring Security que usa GeoServer) → Guardar.
2
Pestaña Usuarios/Grupos → Agregar nuevo usuario
Nombre: editor_gis · contraseña propia · casilla "Habilitado" marcada.
3
Asignarle el rol en la misma pantalla
En "Roles disponibles" selecciona ROLE_EDITOR y muévelo a "Roles asignados" → Guardar.
4
Probar el login en una ventana de incógnito
Entra como editor_gis: ve la interfaz de administración, pero sin poder crear ni borrar nada — todavía no tiene ninguna regla que le dé permisos especiales. Eso lo hacemos en el siguiente paso.
Paso 3 de 5
Reglas de acceso
a los datos
Workspace público · capa privada · escritura restringida
Data security
La sintaxis de las reglas: workspace.capa.permiso = roles
Las tres reglas típicas de un proyecto
# Todo el workspace es visible para cualquiera (lectura) bolivia_curso.*.r = ROLE_ANONYMOUS # Una capa concreta solo para editores bolivia_curso.v_altiplano.r = ROLE_EDITOR # Escribir (WFS-T) solo administradores bolivia_curso.*.w = ADMIN
.rread — ver la capa (WMS, WFS de lectura)
.wwrite — modificar datos vía WFS-T
*comodín — todas las capas del workspace
ROLE_ANONYMOUScualquiera, sin iniciar sesión
⚖️
La regla más específica gana
bolivia_curso.v_altiplano.r le gana a bolivia_curso.*.r. Por eso puedes tener el workspace abierto y una sola capa protegida al mismo tiempo.
⚠️
Si no escribes ninguna regla
Un workspace sin reglas de datos queda abierto por defecto. Declara siempre una regla explícita para bolivia_curso — aunque sea "todos pueden leer" — para que quede documentado y no dependa del comportamiento por defecto.
Práctica
Proteger v_altiplano — la vista de la Clase 6, ahora privada
1
Seguridad → Datos → Agregar nueva regla
Workspace: bolivia_curso · Capa: v_altiplano · Modo de acceso: Read.
2
Asignar el rol ROLE_EDITOR a la regla
Solo quien tenga ese rol podrá ver la capa. Guardar.
3
Probar sin sesión (incógnito)
Layer Preview de v_altiplano → GeoServer responde error de seguridad o pide login. Las demás capas siguen visibles con normalidad.
4
Probar como editor_gis
Iniciar sesión con editor_gis → la capa vuelve a verse. La regla funciona en ambas direcciones: bloquea a quien no tiene el rol, deja pasar a quien sí lo tiene.
✓
Este es exactamente el patrón de cualquier IDE real: mapa base público para todos, capas sensibles solo para usuarios autorizados.
Paso 4 de 5
Reglas de
servicios
Qué operación puede ejecutar cada rol
Service security
Datos vs. servicios — no son la misma regla
Seguridad → Servicios — reglas servicio.operación = roles
# Descargar datos (WFS) solo autenticados wfs.GetFeature = ROLE_AUTHENTICATED # Transacciones WFS-T solo administradores wfs.Transaction = ADMIN # WMS sin regla propia: el mapa queda público
🧭
Qué controla cada una
Las reglas de datos (paso anterior) dicen qué capas ve cada rol. Las reglas de servicios dicen qué operaciones puede ejecutar. Se combinan: para descargar una capa privada por WFS hay que pasar las dos verificaciones.
🎯
Caso de uso típico
Con wfs.GetFeature = ROLE_AUTHENTICATED cualquiera ve el mapa (WMS sigue abierto), pero solo quien inicie sesión puede descargar los datos crudos por WFS. Visualización pública, descarga controlada.

Backup del Data Directory
Copia de /opt/geoserver_data/ en almacenamiento externo, con el servidor detenido o con una herramienta de backup en caliente.
⚠️
GeoServer 3 — cambios frente a GS 2.x
H2 eliminado como base de datos interna → migrar a PostGIS. CORS en web.xml ya no funciona, porque Tomcat/Jetty 12 lo maneja distinto. GEOSERVER_LOG_LOCATION solo por variable de entorno. Keycloak → extensión OIDC.
🔒
geoserver_user sigue siendo la última defensa
Aunque falle todo lo anterior, el usuario de PostGIS que usa el Store (Clase 3) solo tiene SELECT. Ningún atacante que solo comprometa GeoServer puede borrar tus datos en la base.
Resumen
Lo que aprendiste hoy
  • admin/geoserver es la primera credencial que prueban los bots — cambiarla es lo primero
  • El modelo tiene tres piezas: usuarios se autentican, roles agrupan, reglas autorizan
  • Las reglas de datos siguen la sintaxis workspace.capa.permiso = roles, y la más específica gana
  • Las reglas de servicios controlan operaciones: wfs.GetFeature, wfs.Transaction
  • La ventana de incógnito es tu herramienta de prueba: lo que ve un anónimo, lo ve el mundo
  • La master password protege el keystore — es distinta de la contraseña de admin
  • El checklist final cubre CORS, GeoWebCache, logs y backup — la capa de infraestructura
