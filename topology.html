Parte 1
Vistas SQL:
capas sin duplicar datos
Una consulta guardada que GeoServer publica como si fuera una tabla
Concepto clave
¿Qué es una vista SQL?
🗄️
Tabla física
Los datos viven en disco. departamentos tiene las 9 filas con sus geometrías. Ocupa espacio y se importó una sola vez.
🪞
Vista SQL
Una consulta guardada con nombre. No copia datos: cada vez que alguien la consulta, PostgreSQL ejecuta el SELECT sobre la tabla original.
✓
Para GeoServer es una tabla más
El Store PostGIS lista tablas y vistas por igual. Publicar una vista es exactamente el mismo flujo que publicar una tabla.
💡
¿Para qué sirve en un mapa?
Filtrar (solo el altiplano), calcular (superficie, densidad), transformar (polígonos → puntos para etiquetas). Una tabla importada, muchas capas publicadas.
Vista 1 · Filtro
v_altiplano — una capa solo con La Paz, Oruro y Potosí
Crear la vista — un filtro sobre la tabla existente
# Crear la vista sudo -u postgres psql -d geoserver_curso -c \ "CREATE VIEW v_altiplano AS SELECT * FROM departamentos WHERE first_nom_ IN ('La Paz','Oruro','Potosi');" # Verificar: debe devolver 3 filas sudo -u postgres psql -d geoserver_curso -c \ "SELECT first_nom_ FROM v_altiplano;"
🔤
Los campos van en minúsculas
En el Shapefile el campo se llama FIRST_NOM_, pero al importarlo con ogr2ogr los nombres se convierten a minúsculas: en PostGIS es first_nom_. Revisa los valores exactos con SELECT first_nom_ FROM departamentos;
✓
Si mañana se corrige la geometría de Oruro en departamentos, la vista se actualiza sola — no hay datos duplicados que mantener.
Vista 2 · Cálculo espacial
v_departamentos_centroides — de polígonos a puntos con superficie
Funciones PostGIS dentro de la vista
sudo -u postgres psql -d geoserver_curso -c \ "CREATE VIEW v_departamentos_centroides AS SELECT ogc_fid, first_nom_, ROUND((ST_Area(wkb_geometry::geography) / 1000000)::numeric, 0) AS superficie_km2, ST_Centroid(wkb_geometry) AS geom FROM departamentos;" # Verificar: Santa Cruz debe salir primero (el departamento más grande) sudo -u postgres psql -d geoserver_curso -c \ "SELECT first_nom_, superficie_km2 FROM v_departamentos_centroides ORDER BY superficie_km2 DESC;"
🧮
Dos funciones espaciales trabajando
ST_Centroid() convierte cada polígono en su punto central — ideal para etiquetas o íconos. ST_Area(::geography) calcula la superficie real en m² (por eso dividimos entre 1 000 000 para tener km²). El cálculo ocurre en el servidor, no en el cliente.
Permisos
Un paso que muchos olvidan: GRANT sobre las vistas nuevas
Dar lectura a geoserver_user sobre cada vista nueva
sudo -u postgres psql -d geoserver_curso -c \ "GRANT SELECT ON v_altiplano TO geoserver_user;" sudo -u postgres psql -d geoserver_curso -c \ "GRANT SELECT ON v_departamentos_centroides TO geoserver_user;" # Opcional: que todo lo que se cree en el futuro # ya nazca con permiso de lectura sudo -u postgres psql -d geoserver_curso -c \ "ALTER DEFAULT PRIVILEGES IN SCHEMA public GRANT SELECT ON TABLES TO geoserver_user;"
🚨
¿Por qué hace falta de nuevo?
El GRANT SELECT ON ALL TABLES de la Clase 3 solo cubrió lo que existía en ese momento. Las vistas creadas hoy son objetos nuevos — sin este GRANT, GeoServer las lista pero falla al leerlas.
🔍
Síntoma típico
La capa aparece en el Store pero el Layer Preview lanza permission denied for view v_altiplano. Ahora ya sabes exactamente qué falta.

Un nivel más
Vistas materializadas — cuando la consulta es pesada
La vista se calcula una vez y se guarda en disco
# Crear vista materializada sudo -u postgres psql -d geoserver_curso -c \ "CREATE MATERIALIZED VIEW mv_centroides AS SELECT * FROM v_departamentos_centroides;" # Cuando los datos de origen cambien, refrescarla: sudo -u postgres psql -d geoserver_curso -c \ "REFRESH MATERIALIZED VIEW mv_centroides;" # Y darle permiso igual que a las demás sudo -u postgres psql -d geoserver_curso -c \ "GRANT SELECT ON mv_centroides TO geoserver_user;"
⚖️
Vista normal vs materializada
Normal: siempre actualizada, se recalcula en cada consulta. Materializada: resultado guardado en disco, rapidísima de leer, pero hay que refrescarla con REFRESH cuando cambian los datos.
✓
Con 9 departamentos la diferencia es invisible. Con millones de filas y JOINs pesados, la materializada marca la diferencia. Es el mismo principio que veremos ahora con GeoWebCache: calcular una vez, servir muchas veces.
Parte 2
GeoWebCache:
teselas en caché
Calcular una vez, servir muchas veces — ahora con imágenes
Concepto clave
¿Por qué WMS puro se queda corto en producción?
🐢
WMS dinámico
Cada petición dibuja el mapa desde cero: leer PostGIS, aplicar el SLD, renderizar el PNG. 200–2000 ms por imagen, y consume CPU y RAM de la VM.
⚡
WMTS con GeoWebCache
El mapa se corta en teselas de 256×256 px que se generan una sola vez y se guardan en disco. Servir una tesela toma 5–20 ms.
✓
Ya viene dentro de GeoServer
GeoWebCache está integrado — no hay que instalar nada nuevo en la VM. Perfecto para nuestros 2 GB de RAM: servir teselas ya generadas es mucho más liviano que renderizar.
🧩
La misma idea que la vista materializada
La materializada guarda el resultado de la consulta; GeoWebCache guarda el resultado del renderizado. Ambas cambian cómputo repetido por lectura de disco.
Configuración
Activar la caché para una capa
1
Abrir la pestaña "Tile Caching" de la capa
Datos → Capas → departamentos → pestaña Tile Caching.
2
Marcar "Create a cached layer for this layer"
Con eso la capa queda disponible vía WMTS y WMS-C.
3
Elegir gridset y formato
Gridset EPSG:4326 (el CRS del curso) y formato image/png.
4
Guardar
La capa aparece en Tile Caching → Tile Layers, lista para el Seed.
GridsetLa rejilla de teselas: define niveles de zoom y CRS
EPSG:4326Coordenadas geográficas — el CRS que usamos en el curso
EPSG:900913Web Mercator — el de Google/OSM, útil para visores web
image/pngFormato de tesela: soporta transparencia para superponer capas
💡
El estado de la caché por capa se ve en Tile Caching → Tile Layers: ahí están los enlaces de vista previa y las acciones Seed/Truncate.
Seed
Pre-generar las teselas de Bolivia
1
Tile Caching → Tile Layers → Seed/Truncate
En la fila de bolivia_curso:departamentos.
2
Configurar la tarea de Seed
Operación: Seed · Gridset: EPSG:4326 · Formato: image/png · Zoom start: 0 · Zoom stop: 8 · 1 tarea.
3
Limitar el Bounding Box a Bolivia
-69.65, -22.90, -57.45, -9.67 — no tiene sentido generar teselas del océano Pacífico.
4
Submit y observar el progreso
La misma página muestra las tareas en ejecución y cuántas teselas faltan.
💾
Cuidado con el zoom en nuestra VM
Cada nivel de zoom multiplica ×4 el número de teselas. Zoom 0–8 sobre Bolivia son unos cientos de teselas (pocos MB). Zoom 0–15 serían millones — y nuestra VM solo tiene 6 GB de disco. Empieza con 0–8.
🕐
¿Y los niveles sin Seed?
No se pierden: la primera vez que alguien los pide, GeoWebCache los genera y los guarda. El Seed solo adelanta el trabajo de los niveles más usados.
Consumir y mantener
Usar la caché — y truncarla cuando algo cambia
WMTS — el servicio de teselas (para visores como OpenLayers o QGIS)
http://192.168.X.X:8080/geoserver/gwc/service/wmts?REQUEST=GetCapabilities
WMS con caché integrada — solo agregar TILED=true
http://192.168.X.X:8080/geoserver/bolivia_curso/wms? SERVICE=WMS&VERSION=1.1.1&REQUEST=GetMap &LAYERS=bolivia_curso:departamentos &SRS=EPSG:4326&BBOX=-69.65,-22.90,-57.45,-9.67 &WIDTH=256&HEIGHT=256&FORMAT=image/png &TILED=true
✂️
Truncate: la caché no se actualiza sola
Si cambias los datos (un REFRESH de la materializada, un ogr2ogr -overwrite) o el estilo SLD de la capa, las teselas guardadas quedan viejas. Tile Layers → Seed/Truncate → operación Truncate borra las teselas de la zona afectada para que se regeneren con el dato nuevo.
Resumen
Lo que aprendiste hoy
  • Una vista SQL es una consulta guardada: crea capas nuevas sin duplicar datos
  • GeoServer publica vistas exactamente igual que tablas — mismo Store, mismo flujo
  • Las vistas nuevas necesitan su propio GRANT SELECT para geoserver_user
  • ST_Centroid y ST_Area generan geometrías y valores calculados en el servidor
  • Las vistas materializadas guardan el resultado en disco y se actualizan con REFRESH
  • GeoWebCache sirve teselas pre-generadas: 5–20 ms vs 200–2000 ms del WMS dinámico
  • Seed pre-genera (zoom 0–8, BBox de Bolivia) y Truncate limpia cuando los datos o el estilo cambian
