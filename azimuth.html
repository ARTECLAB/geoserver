¿Por qué esta clase?
Un GeoServer sin asegurar es un GeoServer de otro
🤖
Los bots no descansan
Cualquier GeoServer expuesto en una red recibe intentos automáticos con admin/geoserver en minutos. Es la credencial por defecto más conocida del mundo GIS.
🗺️
No todos los datos son públicos
Un mapa base puede ser abierto, pero las capas de catastro, salud o infraestructura crítica de una IDE municipal suelen requerir autorización.

🚨
Nunca en producción con admin/geoserver
GeoServer muestra una advertencia en la página de bienvenida mientras la contraseña por defecto siga activa. Esa advertencia debe desaparecer hoy.
🔑
La master password es otra cosa
Protege el almacén de claves interno (keystore) de GeoServer. Se cambia en Seguridad → Contraseñas. Anótala en un lugar seguro: sin ella no se recuperan las credenciales guardadas, como la del Store PostGIS.

Usuarios y roles
El modelo de seguridad de GeoServer
Modelo de seguridad
Usuarios → Roles → Reglas: tres piezas, una idea
👤
Usuarios
Personas o sistemas que se autentican: admin · editor_gis
🎭
Roles
Etiquetas de permiso: ADMIN · ROLE_EDITOR · ROLE_ANONYMOUS
🔐
Reglas de acceso
Qué rol puede leer (r) o escribir (w) en cada workspace y capa
💡
Las reglas apuntan a roles, no a personas
Si mañana entran 5 editores nuevos al proyecto, solo les asignas el rol ROLE_EDITOR — las reglas no se tocan. Por eso nunca se escribe una regla para un usuario individual.
🆕
GeoServer 3: OIDC reemplaza Keycloak
Para autenticación corporativa, la extensión OIDC soporta OAuth2 estándar: Keycloak, Auth0, Google, Azure AD. En este curso usamos los usuarios locales de GeoServer — suficiente para la IDE municipal.

Paso 3 de 4
Reglas de acceso
a los datos
Workspace público · capa privada · escritura restringida
Data security
La sintaxis de las reglas: workspace.capa.permiso = roles
Las tres reglas típicas de un proyecto
# Todo el workspace es visible para cualquiera (lectura) bolivia_curso.*.r = ROLE_ANONYMOUS # Una capa concreta solo para editores bolivia_curso.v_altiplano.r = ROLE_EDITOR # Escribir (WFS-T) solo administradores bolivia_curso.*.w = ADMIN
.rread — ver la capa (WMS, WFS de lectura)
.wwrite — modificar datos vía WFS-T
*comodín — todas las capas del workspace
ROLE_ANONYMOUScualquiera, sin iniciar sesión
⚖️
La regla más específica gana
bolivia_curso.v_altiplano.r le gana a bolivia_curso.*.r. Por eso puedes tener el workspace abierto y una sola capa protegida.
Práctica
Proteger v_altiplano — la vista de la Clase 6, ahora privada
1
Seguridad → Datos → Agregar nueva regla
Workspace: bolivia_curso · Capa: v_altiplano · Modo de acceso: Read.
2
Asignar el rol ROLE_EDITOR a la regla
Solo quien tenga ese rol podrá ver la capa. Guardar.
3
Probar sin sesión (incógnito)
Layer Preview de v_altiplano → GeoServer responde error de seguridad o pide login. Las demás capas siguen visibles.
4
Probar como editor_gis
Iniciar sesión → la capa vuelve a verse. La regla funciona en ambas direcciones.
✓
Este es exactamente el patrón del proyecto final: mapa base público para todos, capas sensibles solo para usuarios autorizados.

🧭
Datos vs servicios
Las reglas de datos dicen qué capas ve cada rol; las de servicios dicen qué operaciones puede ejecutar. Se combinan: para descargar una capa privada hay que pasar ambas.

Resumen
Lo que aprendiste hoy
  • admin/geoserver es la primera credencial que prueban los bots — cambiarla es lo primero
  • El modelo tiene tres piezas: usuarios se autentican, roles agrupan, reglas autorizan
  • Las reglas de datos siguen la sintaxis workspace.capa.permiso = roles
  • La regla más específica gana: workspace abierto + una capa protegida conviven sin problema
  • Las reglas de servicios controlan operaciones: wfs.GetFeature, wfs.Transaction
  • La ventana de incógnito es tu herramienta de prueba: lo que ve un anónimo, lo ve el mundo
  • En GS3 la autenticación corporativa se integra con la extensión OIDC (OAuth2 estándar)
