Workspace y Store
El contenedor y la fuente de datos — siempre van primero
Flujo de publicación
Workspace → Store → Layer — el orden siempre es este
📁
Workspace: bolivia_curso
Agrupa todo el proyecto. Aparece en la URL: /geoserver/bolivia_curso/wms
🗄️
Store: shapefiles_bolivia → apunta a la carpeta con los .shp
🗄️
Store: postgis_bolivia → conexión a la base de datos
🗺️
Layer: departamentos
🗺️
Layer: municipios
🗺️
Layer: vialidad
✓
Un mismo Workspace puede tener múltiples Stores — uno para Shapefiles, otro para PostGIS. Todos comparten el mismo prefijo de URL.
Paso a paso
Publicar departamentos de Bolivia desde un Shapefile
1
Copiar el Shapefile al Data Directory de GeoServer
/opt/tomcat/webapps/geoserver/data/data/bolivia/departamentos.shp — los cuatro archivos (.shp, .dbf, .shx, .prj) juntos.
2
Datos → Stores → Agregar nuevo Store → Shapefile
Seleccionar el archivo departamentos.shp. GeoServer lee el nombre, CRS y Bounding Box automáticamente.
3
Guardar → Publicar Layer
Hacer clic en "Publicar". En la pestaña Data, verificar el CRS (EPSG:4326) y el Bounding Box. En Publishing, asignar el estilo por defecto.
4
Datos → Layer Preview → Abrir con OpenLayers
Deben aparecer los 9 departamentos de Bolivia. Si no se ven, revisar el Bounding Box en la configuración de la capa.

Layer Groups
Varias capas servidas como una sola petición WMS
Layer Groups
¿Qué son y para qué sirven?
💡
Un Layer Group es un mapa compuesto
El cliente pide LAYERS=mapa_bolivia y GeoServer renderiza todas las capas apiladas en una sola imagen PNG. El cliente no sabe cuántas capas hay por debajo.
🗺️
mapa_bolivia (Layer Group)
Lo que pide el cliente
🏔️
imagen_satelital ← fondo
🌊
hidrografia
🛣️
vialidad
🏙️
departamentos ← encima
Crearlo en GeoServer
1
Datos → Layer Groups → Agregar
2
Darle un nombre: mapa_bolivia
3
Agregar capas en orden — abajo es el fondo
4
Guardar y probar en Layer Preview
✓
Las capas del grupo pueden tener estilos diferentes al publicarlas individualmente. El Layer Group asigna el estilo que corresponde a cada rol en el mapa.
Paso 3 de 4
Segunda capa:
provincias
Repetir el flujo completo refuerza el patrón Workspace → Store → Layer
Paso a paso
Publicar provincias de Bolivia desde un Shapefile
1
Copiar provincias.shp al Data Directory
/opt/tomcat/webapps/geoserver/data/data/bolivia/provincias.shp — junto con .dbf, .shx y .prj.
2
Datos → Stores → Agregar nuevo Store → Shapefile
Mismo Workspace (bolivia_curso) que usaste para departamentos — un Workspace puede tener varios Stores de Shapefile.
3
Publicar → verificar CRS y Bounding Box
Igual que con departamentos: confirmar EPSG:4326 y que el Bounding Box cubra Bolivia.
4
Agregar provincias al Layer Group mapa_bolivia
Encima de departamentos, para ver ambos niveles administrativos a la vez.
✓
El mismo flujo, sin sorpresas
Publicar una segunda capa vectorial es idéntico a la primera. Una vez que el patrón Workspace → Store → Layer queda claro, cualquier Shapefile nuevo se publica en minutos.
Resumen
Lo que aprendiste hoy
  • El flujo de publicación siempre es el mismo: Workspace → Store → Layer. Sin excepciones.
  • Un Store de Shapefile apunta a un único archivo .shp. Los cuatro archivos (.shp, .dbf, .shx, .prj) deben estar juntos en la misma carpeta.
  • Layer Groups agrupa capas para servirlas como una sola — el cliente recibe todo el mapa con una petición WMS.
  • El orden en el Layer Group importa: las capas de abajo son el fondo, las de arriba están encima.
  • Un Workspace admite varios Stores de Shapefile — publicar una segunda capa es el mismo flujo, sin sorpresas.
